# MyTracksy PayHere Setup

This file records the production PayHere setup for the Firebase project `tracksy-8e30c`.
Do not commit merchant secret values, app secrets, test cards, or customer payment tokens.

## Production URLs

- App URL: `https://tracksy-8e30c.web.app`
- Admin URL: `https://mytracksy-admin.web.app/admin.html`
- Customer correspondence and refund email: `info@mytracksy.com`
- WhatsApp / phone: `070 373 6555`

## PayHere Dashboard: Domain/App

In PayHere, go to Side Menu > Integrations and add/request approval for the production domain:

- Domain/App: `tracksy-8e30c.web.app`

If MyTracksy moves to a custom domain, add that custom domain in PayHere too and rotate the Firebase
`PAYHERE_MERCHANT_SECRET` secret to the Merchant Secret issued for that domain.

## PayHere Dashboard: Hosted Payments

Use these values for hosted subscription checkout and wallet/card preapproval flows.

### Subscription recurring checkout

- Return URL: `https://tracksy-8e30c.web.app/payment/return?type=subscription`
- Cancel URL: `https://tracksy-8e30c.web.app/payment/cancel?type=subscription`
- Notify URL: `https://asia-south1-tracksy-8e30c.cloudfunctions.net/handleSubscriptionWebhook`
- Currency: `LKR`
- Recurrence: generated by the app as `1 Month` or `1 Year`
- Duration: generated by the app as `Forever`

### Wallet top-up and saved-card preapproval

- Return URL: `https://tracksy-8e30c.web.app/payment/return?type=wallet`
- Cancel URL: `https://tracksy-8e30c.web.app/payment/cancel?type=wallet`
- Notify URL: `https://us-central1-tracksy-8e30c.cloudfunctions.net/handlePayHereWalletWebhook`
- Currency: `LKR`

The app generates PayHere hashes server-side in Cloud Functions. The frontend only receives signed form
fields and posts the customer to PayHere.

## Firebase Secrets

These Secret Manager entries must exist in project `tracksy-8e30c`:

- `PAYHERE_MERCHANT_ID`
- `PAYHERE_MERCHANT_SECRET`
- `PAYHERE_APP_ID`
- `PAYHERE_APP_SECRET`

The non-secret runtime env flag `PAYHERE_CHARGING_ENABLED` defaults to `false` when absent. Keep it false until PayHere
confirms Merchant API IP whitelisting.

Set or rotate them with:

```bash
firebase functions:secrets:set PAYHERE_MERCHANT_ID --project tracksy-8e30c
firebase functions:secrets:set PAYHERE_MERCHANT_SECRET --project tracksy-8e30c
firebase functions:secrets:set PAYHERE_APP_ID --project tracksy-8e30c
firebase functions:secrets:set PAYHERE_APP_SECRET --project tracksy-8e30c
```

After rotating any PayHere secret, redeploy the payment functions:

```bash
firebase deploy --only functions:initPayHerePreapproval,functions:initPayHereSubscription,functions:handlePayHereWalletWebhook,functions:handleSubscriptionWebhook,functions:oneClickTopUp,functions:processAutoReloads,functions:updatePayHereAutoReload --project tracksy-8e30c
```

## Merchant API: One-Click and Auto-Reload

`oneClickTopUp` and `processAutoReloads` use the PayHere Merchant Charging API. In live mode, PayHere
requires API key domain approval and server IP whitelisting for Merchant APIs.

Current Firebase Functions outbound IP is not guaranteed to be static. Before enabling live one-click or
auto-reload charging for customers, choose one of these paths:

1. Hosted-only launch: keep subscription checkout and wallet preapproval/top-up live through PayHere
   hosted pages. This avoids static egress infrastructure.
2. Full automated charging: configure static outbound egress for the payment functions, send the static
   IP to `support@payhere.lk`, wait for PayHere whitelisting, then run a real low-value live charge test.

Do not treat one-click or auto-reload as live-ready until PayHere confirms the Merchant API whitelist.

After PayHere confirms the whitelist, enable automated charging with a local functions env file:

```bash
printf 'PAYHERE_CHARGING_ENABLED=true\n' > functions/.env.tracksy-8e30c
firebase deploy --only functions:oneClickTopUp,functions:processAutoReloads,functions:updatePayHereAutoReload --project tracksy-8e30c
```

`functions/.env*` is ignored by git so the local deploy flag is not committed accidentally.

Expected webhook reachability results for unsigned test posts:

- Wallet webhook: `403 Invalid merchant`
- Subscription webhook: `400 Unknown webhook format`

Those failures are correct because the endpoints are public but fail closed without a valid PayHere payload.
